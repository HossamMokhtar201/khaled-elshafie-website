# خطوات النشر (Deployment)

## 1) قرار الاستضافة — Hostinger Cloud Startup

**القرار النهائي: نشر الموقع كـ Static Export (HTML/CSS/JS ثابت بالكامل)، وليس كتطبيق Node.js يعمل بسيرفر.**

### التحقق الفعلي (حسب طلب البريف قسم 6.1)
لم تتوفر لهذه البيئة (بيئة تنفيذ Claude Code السحابية) صلاحية الدخول الفعلي على حساب Hostinger الخاص بالمشروع أو لوحة تحكمه للتحقق المباشر من حدود خطة "Cloud Startup" (هل تدعم Node.js runtime كامل أم لا). هذا **قرار بشري/حساب فعلي خارج نطاق ما يمكن لأداة برمجية التحقق منه بدون بيانات دخول** — وهو بالضبط نوع القرار الذي ينص البريف (قسم 9) على تنفيذه بأفضل افتراض ممكن وتوثيقه بدل التوقف.

**الافتراض المطبَّق (موثّق كما طلب البريف):**
- خطط Hostinger "Cloud Startup" مصممة أساسًا لاستضافة مواقع WordPress/PHP وتطبيقات مُدارة، ودعمها لتشغيل Node.js كسيرفر طويل الأمد (Long-running Node process, غير Serverless) محدود أو غير مضمون حسب الخطة، بعكس منصات مثل Vercel/Railway المصممة خصيصًا لـ Next.js.
- بما أن هذه المرحلة من المشروع (Frontend فقط، بدون باك إند حقيقي أو قاعدة بيانات) **لا تحتاج فعليًا** أي ميزة تتطلب سيرفر Node (كل المحتوى من ملفات JSON محلية وقت الـ build، لا API routes حقيقية، لا Server Actions، لا ISR)، فإن **Static Export هو الخيار الأنسب تقنيًا بغض النظر عن نتيجة التحقق** — يعمل على أي استضافة (Hostinger Shared/Cloud، Nginx، Apache، أو حتى CDN بسيط) بدون أي تعقيد أو تبعية لدعم Node.
- تم تنفيذ هذا القرار عمليًا منذ Milestone 0 (`next.config.ts` → `output: 'export'`) وتم التحقق من نجاحه طوال كل المايلستونز التالية.

**عند بناء الداشبورد/الباك إند لاحقًا** (مرحلة خارج نطاق هذا البريف)، سيُحسم عندها فعليًا: إما نقل الاستضافة لخدمة تدعم Node الكامل (Vercel/Railway/VPS)، أو الإبقاء على الفرونت Static والباك إند كـ API منفصل (Headless) يُستدعى من المتصفح — الخيار الثاني لا يتطلب أي تغيير في الفرونت إند الحالي.

## 2) بناء الإصدار النهائي

```bash
npm install
npm run build
```

الناتج: مجلد `/out` في جذر المشروع، يحتوي على HTML/CSS/JS ثابت بالكامل لكل الصفحات الـ26 (بما فيها `robots.txt` و`sitemap.xml` كملفات مولَّدة تلقائيًا).

## 3) رفع الموقع على Hostinger

1. من لوحة تحكم Hostinger (hPanel) → File Manager (أو عبر FTP/SFTP باستخدام بيانات الدخول من hPanel → Files → FTP Accounts).
2. ارفع **محتوى** مجلد `/out` (وليس المجلد نفسه) إلى `public_html/` (أو المجلد الجذر لدومين الموقع).
3. تأكد أن `index.html` موجود مباشرة داخل `public_html/` بعد الرفع.
4. **إعداد الروابط النظيفة (Clean URLs)**: الموقع مبني بـ `trailingSlash: false` الافتراضي، فكل صفحة تُصدَّر كملف `اسم-الصفحة.html` (مثال: `about.html`) وليس `about/index.html`. معظم استضافات Apache (المستخدمة في Hostinger) تدعم تلقائيًا خدمة `page.html` عند طلب `/page` عبر `MultiViews`؛ إن لم يعمل ذلك تلقائيًا، أضف ملف `.htaccess` التالي في جذر `public_html/`:

```apache
Options -MultiViews
RewriteEngine On
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME}.html -f
RewriteRule ^(.*)$ $1.html [L]

ErrorDocument 404 /404.html
```

5. اربط الدومين الرسمي من hPanel إذا لم يكن مربوطًا بالفعل، وفعّل SSL المجاني (Let's Encrypt) من نفس اللوحة.
6. حدّث `NEXT_PUBLIC_SITE_URL` (راجع `src/lib/site-url.ts` — حاليًا Placeholder `https://khaled-elshafie.com`، راجع `ASSUMPTIONS.md`) بالدومين الفعلي وأعد البناء قبل الرفع النهائي، حتى يكون `sitemap.xml` والـ Open Graph tags صحيحة.

## 4) معاينة المراجعة (Preview) على Vercel

يُنصح بربط الريبو بحساب Vercel (خارج نطاق أدوات هذا التنفيذ — خطوة تُنفَّذ يدويًا من مالك المشروع عبر vercel.com → Add New Project → استيراد من GitHub) للحصول على معاينة تلقائية (Preview Deployment) مع كل Push، بينما يبقى النشر الرسمي النهائي على Hostinger.

## 5) نتائج فحص الأداء وإمكانية الوصول (Lighthouse)

تم تشغيل Lighthouse فعليًا (عبر Playwright + Chromium) على الإصدار النهائي (`next build` + خادم استاتيكي محلي) لعدة صفحات تمثيلية:

| المحور | النتيجة |
|---|---|
| Accessibility | **100** |
| Best Practices | **100** |
| SEO | **100** |
| Performance (Desktop) | **99** |
| Performance (Mobile, simulated throttling) | 78-86 |

**ملاحظة مهمة حول نتيجة الأداء على الموبايل محليًا:** الفحص محليًا كان ضد خادم استاتيكي بسيط (`serve`) بدون ضغط Gzip/Brotli أو HTTP/2 أو CDN — وهي جميعها متاحة تلقائيًا على أي استضافة إنتاج حقيقية (Hostinger أو Vercel) وتقلل زمن التحميل بشكل ملموس. بالإضافة لذلك، تفصيل مقاييس LCP (`largest-contentful-paint-element`) أظهر أن الوقت الفعلي المقاس لعرض العنصر الأكبر لا يتجاوز ~230ms (33ms وقت أول بايت + 196ms تأخير عرض)، بينما القيمة الإجمالية المُبلَّغة (4+ ثانية) ناتجة عن نموذج محاكاة التقييد الشبكي/المعالج في Lighthouse عند تطبيقه على خادم محلي بزمن استجابة شبه صفري — وهي مشكلة معروفة في أداة Lighthouse عند القياس ضد `localhost`. **يُوصى بشدة بإعادة تشغيل Lighthouse ضد الرابط الفعلي بعد النشر على Hostinger للحصول على رقم نهائي دقيق**، لكن كل المؤشرات الفعلية (حجم الصفحة الإجمالي 715 KB فقط، صفر Layout Shift، صفر-60ms Total Blocking Time، لا صور حقيقية ثقيلة بعد) تؤكد أن الموقع خفيف وسريع بطبيعته.
- تم إزالة مكتبة `framer-motion` بالكامل من المشروع (Milestone 6) بعد أن أصبح استخدامها الوحيد المتبقي (عدّاد الإحصائيات) قابلاً للتنفيذ بـ `IntersectionObserver` الأصلي بنفس الأداء البصري بحجم أقل ودون التبعية الإضافية — إجراء تحسين أداء مباشر بالإضافة إلى أنه يُبسّط الكود (لم تعد الحركة تعتمد على مكتبة خارجية إطلاقًا).
- تقليص أوزان خط العناوين (Noto Kufi Arabic) من 4 أوزان إلى وزنين (700, 800) فقط بعد التأكد أنهما الوزنان الوحيدان المستخدَمان فعليًا في كل العناوين بالموقع.
